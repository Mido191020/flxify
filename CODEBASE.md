## 2. Tools Registry / Data File

There is no single registry file in the traditional sense. Scripts are **auto-discovered** at build time from the `scripts/` directory. Every `.js` file in `scripts/` is a tool.

### Script anatomy вЂ” `scripts/FormatJSON.js` (complete)

```javascript
/**
	{
		"api":1,
		"name":"Format JSON",
		"description":"Formats JSON with proper indentation and validates syntax",
		"author":"Ivan",
		"icon":"broom",
		"tags":"json,prettify,clean,indent"
	}
**/

function main(state) {
	try {
		state.text = JSON.stringify(JSON.parse(state.text), null, 2);
	}
	catch(error) {
		state.postError("Invalid JSON")
	}
}
```

Every script has two parts:

| Part | Purpose |
|------|---------|
| `/** { ... } **/` block | JSON metadata: `api`, `name`, `description`, `author`, `icon`, `tags` |
| `function main(state)` | The transform logic вЂ” receives a `BoopState` instance |

### How scripts become available at runtime

`build_app.js` reads every `.js` file in `scripts/`, strips the metadata comment, and inlines the `main()` body as a real JavaScript function inside `app.js`:

```javascript
// Inside the generated app.js (simplified):
scripts.push({
  api: 1,
  name: "Format JSON",
  description: "Formats JSON with proper indentation and validates syntax",
  icon: "broom",
  tags: "json,prettify,clean,indent",
  execute: function(require, state) {
    try {
      state.text = JSON.stringify(JSON.parse(state.text), null, 2);
    } catch(error) {
      state.postError("Invalid JSON");
    }
  }
});
```

No `eval()`, no `new Function()`, no runtime fetches вЂ” everything is bundled at build time.

## 4. Routing

**There is no client-side router.** Flxify is a single-page application on the homepage and a collection of **statically-generated HTML pages** for SEO.

| URL pattern | Source | Description |
|-------------|--------|-------------|
| `/` в†’ `index.html` | Hand-authored | Main app with full editor |
| `/tools/` в†’ `tools/index.html` | Generated by `build_app.js` | Tool directory (all 110 tools with categories + search) |
| `/tools/<slug>/` в†’ `tools/<slug>/index.html` | Generated by `build_app.js` | Individual tool page (same editor, auto-selects the matching script) |

Each tool page sets `window.flxifyAutoScript = "Script Name"` before `app.js` loads, which causes `app.js` to highlight/pre-run that script. No JavaScript router is involved; navigation between pages is plain `<a href>` links.

Slug generation (from `build_app.js`):

```
"Format JSON"       в†’ format-json
"UUID Generator"    в†’ uuid-generator
"Base64 Encode"     в†’ base64-encode
```

## 5. One Complete Tool Component Example

Every tool is a single `.js` file with a metadata header and a `main()` function. Below is the complete source of `scripts/FormatJSON.js`:

```javascript
/**
	{
		"api":1,
		"name":"Format JSON",
		"description":"Formats JSON with proper indentation and validates syntax",
		"author":"Ivan",
		"icon":"broom",
		"tags":"json,prettify,clean,indent"
	}
**/

function main(state) {
	try {
		// I feel like this should have a real parser/formatter
		// but hey, it works so who am I to judge?
		state.text = JSON.stringify(JSON.parse(state.text), null, 2);
	}
	catch(error) {
		state.postError("Invalid JSON")
	}
}
```

### BoopState API вЂ” what `state` provides

| Property / Method | Type | Description |
|-------------------|------|-------------|
| `state.text` | `string` (read/write) | Smart accessor: reads/writes `selection` if text is selected, otherwise reads/writes `fullText` |
| `state.fullText` | `string` (read/write) | The entire editor document |
| `state.selection` | `string \| null` (read/write) | Currently selected text (null when nothing selected) |
| `state.isSelection` | `boolean` (read-only) | Whether text is currently selected |
| `state.postError(msg)` | function | Display a red error toast |
| `state.postInfo(msg)` | function | Display an info toast |
| `state.insert(text)` | function | Insert text at the cursor position (for generators) |

## 8. Tech Stack Summary

### Framework
**None.** This is a **plain HTML + CSS + vanilla JavaScript** application. There is no React, Next.js, Vue, Angular, Svelte, or any other component framework. The build step (`node build_app.js`) is a custom Node.js script that inlines script files into a self-contained `app.js` bundle.

### UI Library
**None.** All styles are hand-authored in `style.css`. No Tailwind, no MUI, no shadcn, no Bootstrap. The only external visual dependency is CodeMirror 6 (loaded from `esm.sh` CDN).

### Editor
**CodeMirror 6** вЂ” loaded as ES modules from `https://esm.sh`. Provides syntax highlighting, line numbers, bracket matching, undo/redo, and theme support. Exposed as `window.cmEditor` for `app.js` to read/write editor content.

### State Management
**No state management library.** Application state is minimal:
- Editor content persisted to `localStorage` (key: `flxify-editor-content`, 300 ms debounce)
- Active theme stored in `localStorage` (key: `flxify-theme`)
- Script list is a plain JavaScript array (`scripts[]`) built synchronously at page load
- Command palette open/close controlled by CSS class toggling

### How Tools Are Structured
Each of the 110 tools is its own `.js` file in `scripts/`. All files follow the same shape:

```
metadata comment block  в†’  parsed by build_app.js at build time
function main(state)    в†’  inlined into app.js as an execute() function
```

There are **no shared UI components per tool** вЂ” every tool reuses the same single CodeMirror editor. Running a tool means calling its `execute()` function with the current editor state.

### Search
Fuzzy search is implemented from scratch in `app.js`. When the command palette opens, keystrokes filter the `scripts[]` array using a scoring function:

```
score = nameMatch Г— 0.9 + tagsMatch Г— 0.6 + descriptionMatch Г— 0.2
```

Results are sorted by descending score and rendered as `<li>` elements in `#results`. No external search library is used.

### Routing
**File-based static routing only.** The homepage is `index.html`. The 110 individual tool pages under `tools/<slug>/index.html` are generated at build time by `build_app.js`. There is no JavaScript router; links between pages are plain HTML anchors.

### Build Pipeline
```
node build_app.js
  в†’ reads scripts/*.js           (auto-discovered)
  в†’ reads scripts/lib/*.js       (auto-discovered)
  в†’ generates app.js             (bundled, CSP-safe, no eval)
  в†’ generates tools/<slug>/      (110 static SEO pages)
  в†’ generates tools/index.html   (tool directory)
  в†’ generates sitemap.xml
  в†’ generates robots.txt
```

### Testing
**Vitest 3** with `globals: true`. 700+ tests across:
- Bulk metadata validation for all 110 scripts
- Basic execution tests for 106 scripts
- Targeted tests for scripts requiring specific input (JSON, Base64, hashing, JWT, etc.)

Run with `npm test`. Full validation (test + build + syntax check) with `npm run validate`.

### Deployment
Static files only вЂ” no server required. Works from `file://` URLs, any static host (GitHub Pages, Netlify, etc.), or a CDN. First load requires internet access for CodeMirror from `esm.sh`; subsequent visits work offline via the service worker.
